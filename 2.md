## Shared Layer

### PrismaService (`shared/database/`)
Single Prisma client instance. Injected by all modules that need DB access. `PrismaModule` is global-scope.

### FirebaseService (`shared/firebase/`)
Wraps Firebase Admin SDK. Two responsibilities:
1. `verifyIdToken(idToken)` ‚Äî used by customer/driver auth controllers to validate Firebase tokens before issuing JWTs
2. `sendNotification*()` methods ‚Äî used exclusively by `NotificationService`

Nothing outside `NotificationService` and the auth controllers calls `FirebaseService` directly.

### Canonical Pricing (`modules/pricing/canonical-pricing.ts`)
All pricing logic lives here. Single source of truth for `baseFare`, distance slabs, GST, ETA, and capacity. Both order creation and price preview use the same function. Keyed by `City √ó VehicleType`. To add a city: add a new block to `CITY_CANONICAL_PRICING`.

## Authentication Architecture

### Customer Auth
- `POST /customer/auth/firebase` ‚Äî accepts Firebase ID token, verifies via `FirebaseService.verifyIdToken()`, finds or creates `User` record, returns JWT
- JWT payload: `{ userId, name, city }`
- Dev bypass: `POST /customer/auth/dev-login` accepts `{ userId, secret }` ‚Äî must be disabled in production

### Driver Auth
- `POST /driver/auth/firebase` ‚Äî same pattern, finds or creates `Driver` record
- JWT payload: `{ driverId, firebaseUid }`
- Dev bypass: `POST /driver/auth/dev-login`

### Admin Auth
- `POST /admin/auth/login` ‚Äî email + password, validated against DB, returns JWT
- JWT payload: `{ adminId, role: 'admin' }`

### Route Protection
All protected routes use `@UseGuards(AuthGuard('jwt'))`. Role separation is by checking the JWT payload: customer routes expect `req.user.userId`; driver routes expect `req.user.driverId`. There is no NestJS `@Roles()` guard ‚Äî it's implicit via payload field presence.

## Event-Driven Architecture

One event currently exists: `order.created` (defined in `events/order.events.ts`).

```
PaymentsService.finalizePayment()
    ‚îî‚îÄ‚îÄ eventEmitter.emit('order.created', new OrderCreatedEvent(order))
            ‚îî‚îÄ‚îÄ @OnEvent('order.created', { async: true })
                DispatchService.handleOrderCreated(event)
                    ‚îî‚îÄ‚îÄ notifyEligibleDrivers(order)
```

`EventEmitterModule` is configured globally with `{ wildcard: false }`. The event is emitted **after** the DB transaction commits, so the handler always sees a fully persisted order.

Why this exists: `PaymentsService` should not know about `DispatchService`. The event decouples the payment domain from the dispatch domain. Before this, `PaymentsService` directly called `DispatchService`, which created a circular module dependency and required `forwardRef`.

**`forwardRef` was removed** when `DispatchModule` was made event-driven. If you see `forwardRef` introduced again, it is a signal that a module boundary is being violated.

## Dispatch Architecture

Three triggers for driver notification:

| Trigger | When |
|---------|------|
| `@OnEvent('order.created')` | Immediately after payment confirmed |
| `retryDispatch` cron (`*/2 * * * *`) | Every 2 min for CREATED orders with `expiresAt` still in future |
| `expireCreatedOrders` cron (`*/2 * * * *`) | Every 2 min ‚Äî cancels expired CREATED orders, issues refund, notifies customer |

All three paths call `notifyEligibleDrivers(order)` which:
1. Queries eligible drivers: `isOnline=true`, `kycStatus=APPROVED`, matching city + vehicleType, `lastSeenAt` within 30 minutes, `fcmToken IS NOT NULL`
2. Sends FCM via `NotificationService.notifyDriverNewOrder()`
3. Inserts `OrderOffer` record (idempotent ‚Äî `ON CONFLICT DO NOTHING`)
4. Calls `recordDispatch()` to increment `dispatchAttempts` and stamp `lastDispatchAt`

## Notification Architecture

`NotificationService` is the **only** module that calls `FirebaseService.sendNotification*()`. All other modules inject `NotificationService`.

This was introduced to:
- Centralize FCM payload shaping (title, body, data channel, sound config)
- Allow one place to change notification copy or add data fields
- Prevent modules from bypassing the notification layer and calling Firebase directly

Current notification consumers:
- `DispatchService` ‚Üí new order broadcast
- `DriverOrdersService` ‚Üí assignment, pickup, transit, delivery, cancellation notifications (both driver and customer)
- `AdminService` ‚Üí KYC approval/rejection notifications, admin-cancel notification

All FCM calls are fire-and-forget: they run **after** the DB transaction commits, and failures are caught and logged without affecting the order state.

## Cron Jobs

| Cron | Schedule | Service | What it does |
|------|----------|---------|--------------|
| `retryDispatch` | `*/2 * * * *` | `DispatchService` | Re-broadcasts CREATED orders within TTL to eligible drivers |
| `expireCreatedOrders` | `*/2 * * * *` | `DispatchService` | Cancels expired CREATED orders, refunds, notifies |
| `cleanupAbandonedOrders` | `EVERY_10_MINUTES` | `OrdersService` | Cancels PENDING orders older than 15 min (no refund ‚Äî never paid) |

All crons iterate with per-item try/catch ‚Äî one failure never crashes the loop.

## Platform Fee System

Drivers must pay a daily platform fee before they can receive or accept orders. The fee is enforced in two places:
1. `getIncomingOrders()` ‚Äî SQL JOIN checks `DriverPlatformFee` for today's date; returns 0 orders if unpaid
2. `acceptOrder()` ‚Äî explicit `platformFeeService.hasPaidToday()` check before entering the acceptance transaction

Fee is paid via Razorpay (same flow as customer orders). `DriverPlatformFee` table records payment per driver per day with a `@@unique([driverId, feeDate])` constraint preventing double-payment.

## Concurrency Model

**Driver assignment:** uses PostgreSQL row-level locking via:
```sql
UPDATE "Order"
SET status = 'ASSIGNED', driverId = $id, ...
WHERE id = $orderId AND driverId IS NULL AND status IN ('CREATED', 'CONFIRMED')
```
This is the only correct pattern ‚Äî no application-level lock is used.

**Expiry vs. accept race:** both the expiry cron and driver accept use `WHERE status = 'CREATED'` guards. Only one transaction can succeed; the other sees `rowsAffected = 0`.

## Module Dependency Philosophy

- Modules import `PrismaModule` and `NotificationModule` ‚Äî not each other's services (except where necessary)
- `DispatchModule` depends on `NotificationModule` and `PaymentsModule` (for `issueRefund`)
- `DriverModule` depends on `NotificationModule`, `PlatformFeeModule`, `PaymentsModule`
- `AdminModule` depends on `DriverModule`, `NotificationModule`, `PaymentsModule`
- `PaymentsModule` depends on nothing domain-specific ‚Äî it emits events outward
- Circular dependencies: none currently. Introduce `forwardRef` only as a last resort and document why.
